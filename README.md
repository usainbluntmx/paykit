# ⚡ PayKit

> The accountability layer for autonomous AI agent payments on Solana.

PayKit is open-source infrastructure that gives AI agents their own keypair, their own wallet, enforced spend limits, granular capabilities, and verifiable payment history — all at the protocol level. Agents sign their own transactions. No human required.

Built for the **Solana Frontier Hackathon 2026** by [Zero Two Labs](https://github.com/usainbluntmx).

## The Problem

AI agents are becoming autonomous — but they still can't pay for anything without asking a human first. No wallets. No spend limits. No payment history. No accountability.

Today, if an AI agent needs to pay for an API call, compensate another agent for a service, or manage a spending budget — it has to interrupt a human every single time. There is no standard protocol for agents to transact with each other.

**PayKit solves this.**

---

## What PayKit Does

PayKit is **not** a payment processor. It is an **accountability layer**.

When a developer integrates PayKit, their AI agents get:

1. **Their own keypair** — Each agent has a Solana keypair generated by the SDK. The agent signs its own transactions. The owner wallet is never involved in individual payments.

2. **On-chain identity** — Each agent is a unique PDA (Program Derived Address) derived from the agent's own keypair. Permanent, verifiable, permissionless.

3. **Enforced spend limits** — The contract enforces that an agent can never exceed its total budget. Not 1 lamport more. Cannot be bypassed from the SDK or any application layer.

4. **Configurable daily rate limiting** — Daily limits set per agent in basis points (BPS). 1000 BPS = 10%, 500 BPS = 5%. Resets every 24 hours at the protocol level.

5. **Granular capabilities** — A `u16` bitmask stored onchain defines what each agent can do: pay agents, hire agents by tier, transfer SOL, transfer SPL tokens, batch pay. The contract verifies capabilities before every payment.

6. **Tier system** — Agents have a tier (basic, standard, premium). The contract verifies that a sender has the capability to hire a receiver's tier before executing agent-to-agent payments.

7. **Category limits** — Per-category spending limits (compute, data, storage, inference, research, content). The contract enforces the limit per payment category.

8. **Multi-token support** — SOL and SPL token transfers (USDC and any SPL token) between agent wallets, with onchain accountability recorded automatically.

9. **Automatic expiration** — Agents expire after 365 days. Expired agents cannot transact. Renewable by the owner.

10. **Reversible lifecycle** — Agents can be deactivated, reactivated, and closed. On close, rent (~0.003 SOL) is recovered by the owner.

11. **Immutable payment history** — Every payment emits rich onchain events with agent names, amounts, categories, and timestamps. Anyone can audit any agent with just a PDA address.

## Architecture — Agent-Native

PayKit takes an **agent-native** approach. This is the key distinction from every other agent payment system:

```
Other SDKs:
  Owner signs every payment → agents have no autonomy

PayKit (agent-native):
  Agent has its own keypair → agent signs its own transactions
  Owner only registers and configures → never signs individual payments
```

The PDA is derived from the agent's own keypair:

```
PDA = [b"agent", agent_keypair.pubkey, agent_name]
```

This means the agent's identity is cryptographically tied to its keypair — not to its owner. The owner controls configuration (capabilities, spend limits, tier), but the agent operates autonomously.

## How It Works

```
1. Developer creates an autonomous agent
   → SDK generates a Solana keypair
   → Keypair saved to ~/.paykit/agents/<n>.json
   → Agent PDA registered onchain with spend limit, BPS, capabilities, tier
   → Agent wallet funded with SOL for transaction fees
   → Owner signs this registration — only this once

2. Agent executes tasks autonomously
   → record_payment: logs payment onchain with memo + category
   → transferSOL: moves SOL between agent wallets, agent signs
   → transferUSDC / transferSPL: token transfers, agent signs
   → Spend limit, daily BPS, and category limits enforced by contract
   → Owner is not involved

3. Agent pays another agent
   → agentToAgentPayment: sender agent signs autonomously
   → Contract verifies: capabilities, tier compatibility, spend limits
   → Both agents' counters updated atomically onchain
   → Rich event emitted with names, amounts, category, timestamps

4. Orchestrator pays multiple agents
   → batchPayment: up to 5 payments in one TX, sender agent signs
   → Atomic — all succeed or all fail

5. Anyone monitors in real time
   → watchAgent: polling (development/testing, no external service)
   → createWebhook: Helius webhook to your HTTP endpoint (production)
   → getAgentHistory: agent-specific TX history from onchain
   → /network: live network explorer showing all agents on Devnet

6. Anyone audits
   → Any PDA address → full agent state + payment history
   → No API key, no permission, no database

7. Developer closes an agent
   → closeAgent: closes the PDA, recovers ~0.003 SOL rent
   → Local keypair file removed automatically
```

### Instructions

| Instruction | Signer | Description |
|---|---|---|
| `register_agent` | owner | Creates agent PDA with keypair, spend limit, BPS, capabilities, tier, funding |
| `record_payment` | agent_key | Logs a payment. Enforces spend limit, daily BPS, and category limit |
| `agent_to_agent_payment` | agent_key | Payment between agents. Verifies capabilities and tier compatibility |
| `batch_payment` | agent_key | Up to 5 A2A payments in a single transaction |
| `set_capabilities` | owner | Updates the capabilities bitmask |
| `set_tier` | owner | Updates the agent tier (0=basic, 1=standard, 2=premium) |
| `set_category_limit` | owner | Sets max spend per payment category |
| `set_custom_capability` | owner | Defines a custom capability in bits 8–15 |
| `update_spend_limit` | owner | Updates the total spend limit |
| `deactivate_agent` | owner | Disables agent. Reversible |
| `reactivate_agent` | owner | Re-enables a deactivated agent |
| `renew_agent` | owner | Extends expiration by specified seconds |
| `close_agent` | owner | Closes PDA and recovers rent (~0.003 SOL) |

### Agent Account Schema

```rust
pub struct AgentAccount {
    pub agent_key: Pubkey,                    // Agent's own keypair (signs payments)
    pub owner: Pubkey,                        // Developer wallet (admin operations)
    pub name: String,                         // Unique identifier (max 32 chars)
    pub spend_limit: u64,                     // Maximum total spend in lamports
    pub total_spent: u64,                     // Cumulative spend in lamports
    pub payment_count: u64,                   // Number of payments recorded
    pub is_active: bool,                      // Active/inactive status
    pub bump: u8,                             // PDA bump seed
    pub last_payment_at: i64,                 // Unix timestamp of last payment
    pub daily_spent: u64,                     // Amount spent today in lamports
    pub daily_reset_at: i64,                  // Unix timestamp of last daily reset
    pub expires_at: i64,                      // Unix timestamp of expiration
    pub daily_limit_bps: u16,                 // Daily limit in BPS (1–10000)
    pub capabilities: u16,                    // Permission bitmask
    pub tier: u8,                             // 0=basic, 1=standard, 2=premium
    pub category_limits: [(u8, u64); 8],      // (category_id, max_lamports) × 8
    pub custom_capability_names: [[u8; 16]; 8], // Names for custom cap bits 8–15
}
```

### Capabilities Bitmask

| Bit | Constant | Description |
|---|---|---|
| 0 | `CAN_PAY_AGENTS` | Can pay other registered agents |
| 1 | `CAN_HIRE_BASIC` | Can hire tier-0 (basic) agents |
| 2 | `CAN_HIRE_STANDARD` | Can hire tier-1 (standard) agents |
| 3 | `CAN_HIRE_PREMIUM` | Can hire tier-2 (premium) agents |
| 4 | `CAN_TRANSFER_SOL` | Can transfer SOL between agent wallets |
| 5 | `CAN_TRANSFER_SPL` | Can transfer SPL tokens (USDC, etc.) |
| 6 | `CAN_BATCH_PAY` | Can send up to 5 payments in one TX |
| 8–15 | Custom slots | Owner-defined capabilities |

### Category IDs

| ID | Name | ID | Name |
|---|---|---|---|
| 0 | none | 4 | inference |
| 1 | compute | 5 | research |
| 2 | data | 6 | content |
| 3 | storage | 8–255 | custom |

### Error Codes

| Code | Description |
|---|---|
| `NameTooLong` | Agent name exceeds 32 characters |
| `InvalidSpendLimit` | Spend limit must be greater than zero |
| `InvalidAmount` | Payment amount must be greater than zero |
| `SpendLimitExceeded` | Payment would exceed the agent's total spend limit |
| `AgentInactive` | Agent has been deactivated |
| `MemoTooLong` | Memo or service description exceeds 64 characters |
| `DailyLimitExceeded` | Payment would exceed the agent's daily BPS limit |
| `AgentExpired` | Agent has expired. Use `renewAgent` |
| `InvalidDailyLimit` | `dailyLimitBps` must be between 1 and 10000 |
| `CapabilityDenied` | Agent does not have the required capability |
| `TierNotAllowed` | Agent cannot hire agents of this tier |
| `InvalidTier` | Tier must be 0, 1, or 2 |
| `CategoryLimitExceeded` | Payment exceeds category limit |
| `InvalidCategory` | Invalid category ID |
| `CategorySlotsFull` | All 8 category slots are in use |
| `InvalidCapabilitySlot` | Capability slot must be 0–7 |

### Quickstart — Agent-Native

```javascript
const { createClient, CAPABILITIES, CAP_ALL_DEFAULT, CATEGORIES } = require("@paykit/sdk");

const client = createClient("/path/to/keypair.json", "devnet");

// Create an autonomous agent — generates keypair, registers onchain, funds wallet
const { agentPDA, agentPublicKey, keypairPath } = await client.createAutonomousAgent(
    "my-agent",
    1_000_000_000,   // 1 SOL spend limit
    1000,            // 10% daily limit (1000 BPS)
    50_000_000,      // fund with 0.05 SOL for fees
    CAP_ALL_DEFAULT, // all 7 default capabilities
    1                // tier: standard
);
// Keypair saved to ~/.paykit/agents/my-agent.json

// Record a payment — agent signs autonomously
await client.recordPayment("my-agent", 1_000_000, recipientPubkey, "API call", CATEGORIES.COMPUTE);

// Transfer SOL between agent wallets — agent signs
await client.transferSOL("my-agent", "other-agent", 0.001, "service fee");

// Transfer USDC — agent signs
await client.transferUSDC("my-agent", "other-agent", 1.5, "data analysis");

// Agent-to-agent payment — contract verifies capabilities + tier
await client.agentToAgentPayment("my-agent", "executor", 250_000, "inference task", CATEGORIES.INFERENCE);

// Batch payment — up to 5 agents in one TX
await client.batchPayment("orchestrator", [
    { receiverName: "agent-a", amountLamports: 100_000, service: "research",  categoryId: CATEGORIES.RESEARCH },
    { receiverName: "agent-b", amountLamports: 150_000, service: "writing",   categoryId: CATEGORIES.CONTENT },
]);

// Configure capabilities (owner only)
await client.setCapabilities("my-agent", CAP_ALL_DEFAULT);
await client.setTier("my-agent", 1);
await client.setCategoryLimit("my-agent", CATEGORIES.INFERENCE, 500_000_000);
await client.setCustomCapability("my-agent", 0, "can-access-market-data", true);

// Read agent state
const agent   = await client.fetchAgent("my-agent");
const agents  = await client.fetchAllAgents();
const expiry  = await client.checkAgentExpiry("my-agent");
const balance = await client.getSOLBalance("my-agent");
const history = await client.getAgentHistory("my-agent", 20);
const local   = client.listLocalAgents();
const decoded = client.decodeCapabilities(agent.capabilities);

// Monitor
const stop = client.watchAgent("my-agent", (err, entry) => {
    if (entry) console.log(entry.type, entry.tx);
});
stop();

// Helius webhook (production)
const { webhookId } = await client.createWebhook("my-agent", "https://your-api.com/wh", process.env.HELIUS_API_KEY);
await client.deleteWebhook(webhookId, process.env.HELIUS_API_KEY);

// Close agent and recover rent
const { tx, rentRecovered } = await client.closeAgent("my-agent");
// rentRecovered: "~0.003 SOL" — keypair file also removed locally
```

### Granular Error Handling

```javascript
const { PayKitError } = require("@paykit/sdk/errors");

try {
    await client.agentToAgentPayment("my-agent", "executor", 250_000, "task", CATEGORIES.COMPUTE);
} catch (e) {
    if (e instanceof PayKitError) {
        switch (e.code) {
            case "SpendLimitExceeded":    /* agent hit total budget */ break;
            case "DailyLimitExceeded":    /* daily BPS limit — resets in 24h */ break;
            case "CapabilityDenied":      /* missing required capability */ break;
            case "TierNotAllowed":        /* cannot hire this receiver's tier */ break;
            case "CategoryLimitExceeded": /* exceeds category limit */ break;
            case "AgentExpired":          await client.renewAgent("my-agent", 31_536_000); break;
        }
    }
}
```

## How PayKit Compares

**vs x402 (Coinbase):** x402 handles HTTP payment requests on Base/EVM. PayKit provides onchain agent identity with enforced budgets, capabilities, and immutable history on Solana — 50x cheaper, 30x faster. Complementary: x402 handles HTTP protocol, PayKit handles accountability.

**vs Ethereum AI Agent EIPs (ERC-7715, ERC-7579):** Ethereum delegates permissions from human wallets to agents. PayKit gives agents their own keypair and identity — capabilities, tier, category limits enforced at the protocol level, not the application level.

**vs centralized payment APIs:** No API key, no database, no server to maintain. PayKit is a smart contract on Solana. It cannot be shut down, rate-limited, or censored.

